# Session 9 — 2026-03-10

## Capability Built

**Real integration architecture** — replaced the dead event_loop.py with two
working integration mechanisms: periodic_registry (scheduled tasks run by
ArchiDaemon) and command_registry (on-demand Discord !commands). Capabilities
now actually run when wired.

## What Was Done

### New kernel modules
- Built `src/kernel/periodic_registry.py` — JSON-based registry for periodic
  tasks. ArchiDaemon loads entries at startup and launches an asyncio task for
  each. Entries specify module, coroutine function, and interval.
- Built `src/kernel/command_registry.py` — JSON-based registry for on-demand
  Discord commands. discord_listener matches `!command` messages against
  registered entries and dispatches to the handler. Supports factory patterns
  (`get_scanner().scan_coro`), longest-prefix matching, and `!help`.

### ArchiDaemon changes (run.py)
- Added `_start_periodic_tasks()` method that loads periodic_registry.json,
  resolves each entry's coroutine, and creates asyncio tasks.
- Periodic tasks now launch alongside message + generation tasks at startup.

### discord_listener changes
- Added COMMAND intent — highest priority after IMAGE.
- `_handle_command()` matches `!command` messages, resolves the handler via
  command_registry, and dispatches (async or sync-in-executor).

### event_loop.py deprecation
- Replaced capabilities/event_loop.py with a deprecation stub. All classes
  and functions are no-ops that log warnings. Existing imports won't crash.

### Planner prompt updates (generation_loop.py)
- Added INTEGRATION RULE to both PLAN_SYSTEM and GENERATE_SYSTEM: explains
  periodic_registry and command_registry as the two valid integration targets.
  Explicitly forbids event_loop, wire_*.py, and integrate_*.py.
- Rewrote `_check_reachability()` to check periodic_registry.json and
  command_registry.json in addition to file references. No longer generates
  `wire_` gaps — generates `register_` gaps with proper guidance instead.
- Added periodic_registry.py and command_registry.py to PROTECTED_FILES.

### Data seeding

- Created `data/command_registry.json` with 6 command entries.

### Tests

## What Failed

Nothing. Clean session.

## Design Decisions

- Two separate registries (periodic vs on-demand) for clarity.
- JSON data files, not Python modules — Archi can write entries safely.
- Deprecation stub for event_loop.py rather than deletion (backward compat).
- `register_` gap prefix instead of `wire_` to break the infinite loop.
- Did NOT rewrite the 31 existing capability files — their event_loop imports
  now hit no-op stubs. Archi will rebuild them with proper registration over time.
